servlet - 
    server side java program
    it runs within a servlet container (or web container like Tomcat) which is part of a web server
    it is a generic server extension/class, that can be loaded dynamically to expand the functionality of server
    runs inside JVM , thus safe and portable
    operates only within domain of server
    unlike CGI common gateway interface (uses multiple processes for each program), servlets are all handled by seperate threads, thus they are efficient and scalable

    Servlet	vs Java Program 6
        Runs inside a web/servlet container
            Runs independently
        Handles client requests (HTTP)
            Does not handle client requests
        Uses threads for multiple users
            Single execution flow
        Operates only within server domain
            Runs on local machine
        More efficient than CGI
            No built-in web support
        Must implement Servlet interface
            typically extends Object class and is general purpose
    
    Servlets are specialized Java programs designed for web applications, 
    unlike normal Java programs which run independently and do not handle client-server communication.

    adv of servlets (multi-threading diagram)
        webcontainer creates threads for multiple users, threads have benefit cause they have common memory area, lightweight, low communication cost etc
        better performance : cause thread is created for each request, not process
        portablity as it uses java
        secure as it uses java
        robust : servlets managed by JVM, so no tension for memory leak, garbage collection etc

    Life Cycle - 
        diagram

        initialization
            servlet engine(web container like tomcat) loads servlet class into JVM, servlet class maybe in local or remote system

            doGet(), doPost() etc methods are part of it
        destroy()
            when admin wants to unload servlet, or when system is to shut, servlet engine calls the destroy method
            now the servlet is eligible for garbage collection
            run only once
            common tasks - 
                synchronises cleanup tasks closing resources, connection pools
                informs other applications that server will no longer be in service

9 points ***
| **Property**                    | **include() Method**                                 | **forward() Method**                        |
| ------------------------------- | ---------------------------------------------------- | ------------------------------------------- |
| **What can be done?**           | Includes another file in the current file            | Forwards client request to another resource |
| **Merge of response**           | Responses of S1 and S2 are merged and sent to client | No response merge; only S2 response is sent |
| **Retaining execution control** | Control shifts temporarily from S1 to S2             | Control shifts permanently from S1 to S2    |
| **Control coming back**         | Control returns to S1 after executing S2             | Control never returns to S1                 |
| **Response placement**          | Response of S2 is placed inside response of S1       | Response of S2 replaces S1 response         |
| **Client receives**             | Combined response of S1 and S2                       | Only response of S2                         |
| **Extra activity**              | Further server-side processing is possible (by s1)   | No further processing possible              |
| **Usage**                       | Used when output of both S1 and S2 is required       | Used when only S2 output is required        |
| **Speed of delivery**           | Comparatively slower                                 | Faster                                      |

Session - 
    HTTP is stateless protocol, thus every request is new and independent for server
    server does not remember user from previous request
    
    To solve this, Session object (HttpSession interface) is used
    It is a server side object, used to maintain state of a user accross multiple request
    essentially, creates conversation between browser and server
    exists from login till logout, or browser closed, or timeout
    session attributes available to only single user (like servletConfig)

    flow - 
        client sends login details (username, password)
        server validates, and creates session object in memory to hold user data
        server generates unique session ID, sent back to client , stored in thier browser's cookies
        client makes new request -> automatically sends session ID with request to the server
        server compares session ID with its data stored in memory to identify user and retireve their specific data
        sends that data back to client
    
    Code :
    public class SessionExample extends HttpServlet {

        public void doGet(HttpServletRequest request,
                        HttpServletResponse response)
                        throws IOException {

            response.setContentType("text/html");   
            PrintWriter out = response.getWriter();

            // get session or create one
            HttpSession session = request.getSession(true);

            // Session details
            String sessionId = session.getId();
            long creationTime = session.getCreationTime();

            // Store data in session
            session.setAttribute("user", "Anuj");

            // Display output
            out.println("Session ID: " + sessionId);
            out.println("<br>Session Created At: " + new Date(creationTime));
        }
    }

response.sendRedirect() - ***

    redirection to external websites is possible, unlike forward()

        client sends request to first servlet (s1)

        thus, server sends response back to browser with specific status code and header indicating location of new URL
        browser receives response, reads new url and generates new fresh call to new location
        new resource processes this req, and sends final response to user
    
    as browser makes new fresh call , original res and req objects are destroyed after first part

    A Filter intercepts a request before it reaches the servlet and/or a response before it is sent back to the client.
    Filters are executed in a chain defined by the deployment descriptor (web.xml).
    The server (web container):
        Calls init() once when the filter is loaded
        Calls doFilter() for every request
        Calls destroy() when the filter is removed

JDBC - 
    Java Database Connectivity
    java API providing standard interface for connecting Java apps to relation DBs (Mysql, Oracle, PostgreSql)

    Type 1 : JDBC-ODBC bridge driver
        java app -> jdbc api -> odbc driver -> DB
        slowest , deprecated

    Type 2 : Native API (Partly java driver)
        java app -> JDBC api -> Native DB library -> DB
        Faster than type 1, but OS-dependent

    type 3 : Network Protocol Driver (middleware)
        java app -> jdbc api -> middleware server -> DB
        DB-independent , but needs middleware
    
    type 4 : Native Protocol driver (thin driver)
        java app -> jdbc api -> direct DB (tcp/ip)
        fastest, production grade

    Type 1 (JDBC-ODBC bridge driver)
        converts JDBC calls into ODBC calls 
        uses ODBC driver installed on client machine

        diagram - 
        Java App ---------> JDBC Type 1 driver ---------------> ODBC Driver ------------> DB (e.g MS access)
                 jdbc calls                     odbc calls                    db protocol

        adv - 
            easy to use (built into JDK)
            supports any ODBC enabled DB
            good for dev/testing
        disadv - 
            slowest of all drivers
            requires odbc on client
            not good for production

    Type 4 (Native Protocol driver)
        thin driver
        converts JDBC calls into DB specific protocols
        communicates directly over network without any middleware
        java app -> jdbc driver (100% java) -> direct DB (tcp/ip)

1) Difference between JSP and servlet.
    JSP is a server-side technology 
    used to create dynamic, platform-independent web pages
    embeda Java code into HTML using special tags

    Problems with Servlets (Why we need JSP)
        Code Clutter: Contains request processing, business logic, and UI (response generation) in one file.
        High Learning Curve: Requires deep Java knowledge to develop and maintain.
        Rigid UI: Changing the "look and feel" requires modifying and recompiling Java code.
        Tool Incompatibility: Standard web design tools (Dreamweaver, etc.) cannot easily edit Java files.
        Developer Overhead: Involves writing massive amounts of println statements for HTML.
        Maintenance: Any change or new addition often requires a server restart.

    Advantages of JSP
        Separation of Concerns: * Logic: Handled by Java/JSP elements.
        Presentation: Static HTML remains as-is.
        Role Division: Allows Java programmers to focus on logic while Web authors focus on UI
        Flexibility: Can change the presentation layer without affecting the business layer (and vice versa).
        Efficiency: More convenient to write regular HTML than using out.println() for every tag.

2) Explain life cycle of JSP
JSP Life Cycle
    The sequence of steps a JSP page goes through from the moment it is first requested until it is destroyed
    Managed by: The JSP/Servlet Container (e.g., Tomcat)

    life cycle first diagram

    1 Translation & compilation-
        happens only during the very first request for a specific JSP page
        Translation: 
            JSP engine parses .jsp file for syntactical correctness 
            converts it into a Servlet source file (.java)
            It skips JSP comments and brings in any included files, scriplets etc into the _jspService method
        compilation: 
            source file compiled into servlet class file using standard javac compiler
        
    2 Loading & initialization -
        container loads compiled class file into memory
        instance of this servlet is created
        Initialization: 
            container calls jspInit() once to initialize variables and resources=[s]
            ServletConfig and ServletContext objects become available during this step

    3 Request Processing (Execution)
        _jspService Method:
            heart of JSP life cycle
            equivalent to the service() method in standard servlets.
            called for every single request
            takes HttpServletRequest and HttpServletResponse as parameters
            Note: This method cannot be overridden within a JSP page

    4 Cleanup & Destruction
        jspDestroy Method:
            container calls it before the JSP instance is taken out of service
            guaranteed to be the last method called
            Used to release resources acquired in jspInit() (e.g., closing database connections)



3) what are the implicit objects in JSP
    Variables that can be used in a JSP page without being declared first.
    These objects are made available to the page by the _jspService method

    request:
        HttpServletRequest associated with the current request
        Allows to view request parameters (via getParameter), request types (GET, POST), and HTTP headers (cookies, etc.).

    response:
        HttpServletResponse associated with the client response
        output stream for this object is buffered

    out:
        buffered version of PrintWriter called JspWriter
        Used to send output directly to the client
    
    session:
        HttpSession object associated with the request
        This variable is bound even if there was no incoming session reference, as sessions are created automatically.

    page:
        Acts as a synonym for 'this' in Java.
        serves as a placeholder for potential future scripting languages other than Java.


4) Discuss any 5 JSP elements with examples
    building blocks used to insert Java code or perform specific actions within a JSP page.

    1. JSP Expressions
        Evaluates a Java expression and inserts the resulting string directly into the output.
        Syntax: <%= expression %>.
        Example:
        Today is: <%= new java.util.Date() %>

    2. JSP Scriptlets
        Allows you to insert snippets of Java code into the servlet's _jspService method.
        Syntax: <% code %>.
        Example:
        <% if (Math.random() < 0.5) { 
            out.print("Have a nice day!"); 
        } %>
        (Used for local logic like conditional statements) .

    3. JSP Declarations
        to declare methods or global variables that are placed outside the service method in the servlet class.
        Syntax: <%! code %>.
        Example:
        <%! int count = 0; %>
        (This creates a global variable count available to the entire class) .

    4. Page Directive
        Defines page-specific attributes such as language, imported classes, or error handling.
        Syntax: <%@ page attribute="value" %>.
        Example:
        <%@ page language="java" import="java.util.*"%>

    5. Standard Actions (e.g., Forward)
        Performs dynamic actions at runtime, such as shifting the request to another page.
        Syntax: <jsp:forward page="URL" />.
        Example:
        <jsp:forward page="./login.html" />

Struts -
    Open source java framework for creating web applications using MVC2 arch
    Action Based Framework

    Struts v2 - webwork + struts
        implements MVC2 design pattern

    Core Controller: 
        Since Struts 2.1, the main controller is the StrutsPrepareAndExecuteFilter(SPEF bolenge for now) (previously FilterDispatcher)

    browser requests - /mypage.action
    FilterDispatcher or SPEF: entry point that intercepts every request and determines which Action to run
    ActionMapper - maps request to correct action class
    interceptors - automatically apply common tasks like validation, file upload, and parameter mapping before the action executes
    Action method executes
    Results render the output(JSP) back to browser

    diagram (1st wala)

    Request flow - 
        Request: User sends a request for a specific resource (e.g., /mypage.action).
        Mapping: The Container checks web.xml to map the request to the Struts Controller.
        container invokes controller (FilterDispatcher or SPEF)
        Controller & Mapper: that Controller asks the ActionMapper which Action class to use.
        Proxy Creation: The Controller invokes the ActionProxy, which fetches action & interceptor details from struts.xml.
        Invocation: ActionProxy forwards request to ActionInvocation, which triggers the Interceptors followed by the Action method.
        Response: A Result is generated, is sent back to ActionInvocation, converted into an HttpServletResponse, and sent back to the user.


Interceptors - 
    Classes that implement the com.opensymphony.xwork2.interceptor.Interceptor interface 
    to provide pre-processing and post-processing on Action invocations
    Defined in the struts.xml file

    params - Sets the request parameters directly onto the Action class.
    validation - Validates data using an ActionName-validation.xml file or Annotations.
    fileUpload - Automatically handles file uploads when the request contains multipart/form-data
    exception - Catches exceptions and maps them to a specific result (error page)
    logger - Logs the start and end of the execution of an action for tracking purposes
    cookie - Reads specific cookies and values from the local system
    prepare - Used for pre-processing an object in the Action before methods run
    modelDriven - Creates a POJO(Plain Old Java Object) model class and maps request parameters to it, instead of the Action
    execAndWait - Displays an intermediate "loading" page while the Action executes in the background
    workflow - it doesn't perform validation, but checks for validation errors and controls the flow based on errors

The Problem (Why we need it) - 
    In a microservices architecture (eg - a streaming app like Streamix), services are dynamically scaled up or down
    IP addresses and ports change dynamically upon deployment or failure
    Hardcoding URLs of services (eg - UserBilling service trying to call MovieCatalog service) leads to broken connections and rigid architecture



Components & Flow - 
    1) Eureka Server (Service Registry)
        Central Spring Boot application maintaining a registry of all active microservice instances
        Stores the Application Name, IP address, port, and health status


    2) Eureka Client (The Microservices)
        Every individual microservice (MovieService, UserService, etc.) acts as a client
        Upon startup, it registers its own IP and port with the Eureka Server


    3) Heartbeats (Renewal)
        Clients send periodic pings to the Server (default every 30 seconds) to prove they are alive
        If the Server stops receiving heartbeats from an instance, it evicts that instance from the registry (typically after 90 seconds)


    4) Service Discovery & Routing
        When Service A needs to communicate with Service B, it queries the Eureka Server using Service B's application name
        Eureka Server returns the live IP/Port of Service B
        Service A then makes the direct HTTP call (enabling client-side load balancing)

--

Implementation (Spring Boot & Java 21) - 
    Eureka Server Configuration:
        Add `@EnableEurekaServer` annotation on the main Spring Boot application class
        properties:
            `server.port=8761` (default Eureka port)
            `eureka.client.register-with-eureka=false` (tells the server not to register itself as a client)
            `eureka.client.fetch-registry=false`
    Eureka Client Configuration:
        Add `@EnableDiscoveryClient` on the microservice's main class
        properties:
            `spring.application.name=movie-catalog-service` (The name used by other services to find it)
            `eureka.client.serviceUrl.defaultZone=http://localhost:8761/eureka/` (Points the client to the registry)

adv - 
    Decouples service providers from consumers
    Highly available (multiple Eureka servers can sync with each other)
    Seamless horizontal scaling of microservices without manual configuration changes
    Provides out-of-the-box integration with API Gateways and Load Balancers
